# Markdown style: One Sentence Per Line (OSPL)

Write prose in markdown files with **one sentence per line**.
Start every new sentence on its own line, with no trailing whitespace and no manual wrap-width.
Keep a blank line between paragraphs as usual.

Reference: <https://nick.groenen.me/notes/one-sentence-per-line/>.

## Why

OSPL gives us better diffs and reviews without changing what readers see.

- **Diff hygiene.** A reworded sentence touches a single line, so `git diff` and PR review surface exactly the edit instead of a reflowed paragraph.
- **Reviewability.** Reviewers and `gh pr view` line-number comments land on the sentence being discussed, not on whichever wrap-position the previous editor happened to use.
- **Authoring friction.** Contributors don't have to rewrap a paragraph after every edit; editors can keep soft-wrap on and forget about line length.
- **No rendered-output change.** CommonMark joins consecutive non-blank lines within a paragraph with a single space, so GitHub, npm, and docs sites render OSPL prose identically to traditionally-wrapped prose.
  Verify on the rendered page if you're unsure — e.g. <https://github.com/confluentinc/mcp-confluent/blob/main/CONFIGURATION.md>.

## How to apply

- **One sentence, one line.** Period (or `?`, `!`) ends the sentence; the next sentence starts on the next line. No double line breaks inside a paragraph — that creates a new paragraph in the rendered output.
- **Paragraph breaks are still blank lines.** Two newlines = new paragraph, same as always.
- **Blockquotes, list items, table cells: same rule.** Inside a `>` blockquote or a `- ` list item that contains multiple sentences, break after each sentence and indent the continuation to stay inside the block. Table cells stay single-line — tables don't support intra-cell line breaks in CommonMark.
- **Code blocks, YAML/JSON examples, command-line snippets: do not reflow.** OSPL applies to prose, not to fenced code.
- **Long sentences are fine.** OSPL is not a line-length rule; don't break a single sentence across multiple lines just because it's long.
- **Inline links and emphasis don't change.** OSPL is purely about line breaks between sentences, not about markdown syntax inside a sentence.

## Scope

This rule applies to all prose markdown in the repository: `README.md`, `CONFIGURATION.md`, `CONTRIBUTING.md`, `telemetry.md`, files under `docs/`, and similar.

Two pragmatic exceptions:

- **`CHANGELOG.md`** is generated by tooling; do not hand-reflow it.
- **Single-sentence entries** (badges, one-line bullet lists, front-matter values) are already trivially one-sentence-per-line — no action needed.

When editing an existing markdown file that pre-dates this rule, reformat the paragraphs you're touching to OSPL.
Don't reflow the rest of the file as drive-by churn in the same PR.
